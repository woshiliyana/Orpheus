# Design: v0.40 Plus v0.41 Overlay Into Orpheus Docs Truth

Generated by `superpowers:brainstorming` on 2026-04-23
Branch: `main`
Status: APPROVED IN CHAT, WRITTEN FOR REVIEW

## Problem Statement

The repo already contains a coherent `v0.40` docs overlay and an additional `v0.41` increment focused on provider-cost reality, public benchmark reality, and sharper distribution packaging. The challenge is not whether these bundles contain useful content. They do. The challenge is how to absorb them without destroying the current repo’s truth structure, reintroducing already-fixed review regressions, or creating a fresh double-source-of-truth mess.

The integration must preserve three things at once:

1. `v0.40` is the primary baseline for the docs it explicitly owns.
2. Current repo-only truth and support docs that `v0.40` does not replace must remain in place.
3. `v0.41` must land first as an execution-focused increment on top of `v0.40`, with only the parts that should become canonical actually promoted into truth docs.

## Design Goals

1. Keep Markdown as the only canonical truth.
2. Treat `v0.40` as the primary overlay for overlapping docs.
3. Prioritize `v0.41`’s pricing / benchmark / distribution tightening before any broader secondary cleanup.
4. Treat `v0.41` as an incremental tightening layer, not a replacement baseline.
5. Preserve already validated review fixes even when a bundle version differs.
6. Expand the first technical gate to cover English and Spanish output and timing readiness, while keeping the workspace UI English-first.
7. Keep the import bundles as backup artifacts, not living sources of truth.

## Decisions Locked

### 1. Overlay rule

For overlapping files:

- `v0.40` is the base version.
- Any valid review-backed fixes already confirmed in the current repo must be re-applied on top.
- `v0.41` then tightens specific files where it adds genuinely new constraints.

For non-overlapping files:

- If a file exists only in the bundle and belongs to the formal truth or plan system, add it.
- If a file exists only in the current repo, keep it unless the bundle explicitly supersedes it.

### 2. Canonical truth rule

Canonical truth remains Markdown only:

- `docs/prd/prd.md`
- files under `docs/prd/specs/`
- `docs/prd/source-of-truth-index.md`

The following may support decisions but are not canonical:

- `docs/prd/design-2026-04-23-longform-audio-wedge.md`
- `docs/prd/reviews/2026-04-23-competitive-benchmark-longform-audio.md`
- the pricing workbook

### 3. Workbook role

`orpheus_pricing_unit_economics_model_v0.41.xlsx` is a formal planning artifact and should live in the formal Phase 2 plan set, but it is not canonical truth. If the workbook and Markdown diverge, Markdown wins until both are reconciled in the same change.

### 4. Language boundary

The first technical gate expands from English-only output to bilingual output readiness:

- content generation promise: English + Spanish
- subtitle / timing / alignment readiness promise: English + Spanish
- public landing pages: English + Spanish + Chinese allowed
- full workspace UI: English first
- Spanish workspace UI: second priority after the first gate is proven
- Chinese workspace UI: not part of the initial full end-user product promise

This means Phase 2 feasibility work must use both English and Spanish corpus variants, not English alone.

### 5. Priority order

The integration order is intentionally not “apply everything evenly.”

The first priority is:

1. absorb `v0.41` benchmark, provider-cost, and hook-tightening changes
2. lock the EN/ES output + timing boundary in truth and plans
3. only then finish the rest of the `v0.40` overlay cleanup where it does not conflict

## Review Fixes That Must Survive The Overlay

The following are not optional and must remain true after `v0.40` and `v0.41` are absorbed:

1. Deferred tool pages may not carry immediate trial-entry CTAs.
2. The feasibility corpus must be explicit and frozen.
3. The wedge design doc must not appear in `Depends on Truth Docs`.
4. Homepage IA must not front-load `Try Free`.
5. Pricing and design narrative must not sell the old “starter library + flagship voices” bundle as the first buying reason.
6. The first-gate site structure must stay narrower than multilingual SEO + tool-page rollout.
7. Phase 2 must include a basic voice-selection path and may not quietly pass on one hidden internal voice.

## Testing And Validation Requirements

After integration, the docs state must pass these checks:

1. `git diff --check` is clean.
2. No tool-page immediate trial CTA remains in the distribution spec.
3. The feasibility plan still freezes corpus fixtures explicitly.
4. The design note is not listed as truth anywhere.
5. The pricing workbook is referenced as a planning companion, not as truth.
6. `source-of-truth-index.md` explicitly registers the pricing spec.
7. Phase 2 truth and plan docs all agree that:
   - first gate = hosted-first stable audio
   - EN + ES output and timing readiness are in the gate
   - English-first workspace UI remains the default

## Scope Boundaries

This design does not decide implementation details of the audio pipeline itself.

It only decides:

- how `v0.40` and `v0.41` should be absorbed into the docs system
- which files become canonical truth
- which files become formal planning support
- which review fixes are non-negotiable
- how language scope is represented at the docs / plan layer

## Risks

1. If `v0.41` language expansion is promoted into truth docs without adjusting Phase 2 corpus and review criteria, the docs will over-promise.
2. If the workbook is treated informally, pricing truth will drift again.
3. If review fixes are not re-applied after overlay, the old trial-first funnel and corpus ambiguity will silently return.
4. If the import bundle itself is kept as a live truth source, agents will start reading two sources again.

## Recommended Next Step

After this design is approved in writing, create a writing-plans implementation plan that executes the integration in this order:

1. layer `v0.41` truth + benchmark changes first
2. lock EN/ES output + timing and English-first workspace UI
3. apply the rest of the `v0.40` overlay by file ownership
4. add bundle-only new plan/spec files and workbook placement
5. re-apply mandatory review fixes
6. run review + verification
7. commit and push
