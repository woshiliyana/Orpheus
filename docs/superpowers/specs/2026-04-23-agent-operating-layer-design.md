# Design: Orpheus Agent Operating Layer

Generated by `superpowers:brainstorming` on 2026-04-23
Branch: `main`
Status: APPROVED IN CHAT, WRITTEN FOR REVIEW

## Problem Statement

Orpheus now has a fairly mature docs truth stack:

- `docs/prd/*` for product truth
- `docs/roadmap/*` for sequencing
- `docs/plans/*` for execution packets

What it still lacks is a lightweight, repo-native operating layer for agents. Right now the root contains handoff files for a `v0.43` agents-only increment, but those files are not yet integrated into the repo structure. If left as loose root files, they create drift and are easy to ignore. If integrated badly, they could create a second product constitution that conflicts with PRD truth.

The design goal is to add an agent operating layer that improves execution discipline without competing with canonical product truth.

## Design Goals

1. Keep product truth exactly where it is today.
2. Make `AGENTS.md` the repo runtime entrypoint.
3. Add repo-local skills only for Orpheus-specific execution discipline.
4. Preserve `docs/superpowers/*` as supporting and archival material, not runtime authority.
5. Archive the incoming `v0.43` patch / zip / handoff bundle so the root stays clean.
6. Finish in a state where future agents know exactly what to read, what they may change, and what they must not invent.

## Decisions Locked

### 1. Authority order

The final authority order is:

1. `docs/prd/*`, `docs/roadmap/*`, `docs/plans/*`
2. `AGENTS.md`
3. `.agents/skills/*`
4. `docs/superpowers/*`

This means:

- `AGENTS.md` is runtime instruction, not product truth
- repo-local skills are execution helpers, not policy authors
- `docs/superpowers/*` stays available, but it cannot overrule the runtime layer or canonical docs

### 2. What `AGENTS.md` is allowed to do

`AGENTS.md` may define:

- mandatory read order
- non-negotiable execution guardrails
- parallel/subagent policy
- skill routing
- handoff expectations

`AGENTS.md` may not define:

- product scope
- pricing truth
- lifecycle semantics
- language promises
- Phase 2 gate rules beyond pointing agents back to canonical docs

### 3. What repo-local skills are allowed to do

The `.agents/skills/` layer should contain only Orpheus-specific execution helpers:

- `orpheus-truth-router`
- `orpheus-phase2-stable-audio`
- `orpheus-provider-benchmark`
- `orpheus-parallel-lanes`
- `orpheus-doc-sync`
- `orpheus-change-verification`

These skills should:

- route agents to the right canonical docs
- tighten benchmark and review discipline
- constrain parallel work
- enforce doc backfill and verification

They should not:

- redefine product truth
- replace PRD or roadmap ownership
- invent new packaging/language/lifecycle rules

### 4. What stays in `docs/superpowers`

`docs/superpowers/*` remains useful, but its role is explicitly downgraded to:

- design records
- implementation plans
- bundle/import archives
- supporting process notes

It should not act as a second runtime instruction system.

### 5. How to treat the `v0.43` bundle

The incoming `v0.43` content is not a new product-truth package. It is an add-only agent operating increment.

So the correct move is:

- keep the root `AGENTS.md` and `AGENT_PROMPT_PLAYBOOK.md` as the baseline
- add the missing `.agents/skills/*` and `docs/ops/agent-execution-policy.md`
- archive the raw handoff / patch / zip under `docs/superpowers/import-bundles/orpheus-v0.43/`
- do not leave the loose import files in repo root

## Expected Behavior After Integration

1. A new agent entering the repo reads `AGENTS.md` first.
2. The agent is routed from `AGENTS.md` into Orpheus-specific repo-local skills.
3. Those skills point the agent back into `docs/prd/*`, `docs/roadmap/*`, and `docs/plans/*` as the actual truth.
4. `docs/superpowers/*` remains useful as memory and planning support, but no longer competes with the runtime layer.
5. The repo root remains clean; loose import files no longer sit next to the real repo surface.

## Validation Requirements

The final integrated state should satisfy all of these:

1. `AGENTS.md` and `AGENT_PROMPT_PLAYBOOK.md` are tracked in git.
2. `docs/ops/agent-execution-policy.md` exists.
3. All six `orpheus-*` repo-local skills exist under `.agents/skills/`.
4. No runtime file in the new agent layer contradicts current canonical truth.
5. `docs/superpowers/*` still exists, but nothing in the live runtime path depends on it for product meaning.
6. The loose root `v0.43` files are moved into a backup folder and no longer clutter the root.
7. `git status` is clean after commit/push.

## Risks

1. If `AGENTS.md` becomes too long or too opinionated, it will start competing with canonical docs instead of routing to them.
2. If repo-local skills start embedding product rules, Orpheus will grow a hidden second truth system.
3. If the root bundle files are left loose, future agents may read them as live instructions instead of historical import material.
4. If `docs/superpowers/*` is not explicitly de-authorized, some agents may continue to treat it as the primary operating layer.

## Recommended Next Step

After written approval of this spec, write a short implementation plan that:

1. tracks `AGENTS.md` and `AGENT_PROMPT_PLAYBOOK.md`
2. imports the repo-local skills and `docs/ops/agent-execution-policy.md`
3. archives the `v0.43` root files
4. runs a focused review to confirm the new agent layer does not conflict with current PRD / roadmap / Phase 2 packet truth
